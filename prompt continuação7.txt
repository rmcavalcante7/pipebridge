🏗️ ARQUITETURA CONSOLIDADA DO SDK

pipefy/
├── client/         → comunicação HTTP / GraphQL
├── service/        → regras de negócio (FileService, CardService, etc.)
├── integrations/   → execução de chamadas externas (S3, GraphQL raw)
├── models/         → modelos tipados (CardModel, FieldModel)
├── facade/         → interface pública do SDK
├── exceptions/     → sistema padronizado de erros
├── service/file/flows/
│   ├── pipeline/   → steps + step engine
│   ├── rules/      → regras + rule engine
│   ├── config/     → retry + circuit breaker config
│   ├── resilience/ → circuit breaker

---

🔥 ARQUITETURA IMPLEMENTADA (CORE DO SDK)

## 1. RULE ENGINE (Validação)

Criamos um sistema extensível baseado em:

* BaseRule
* RuleEngine
* Prioridade de execução
* Suporte a regras customizadas pelo usuário

Fluxo:

request → rules → validação → falha ou continua

✔ validação desacoplada
✔ extensível
✔ orientado a domínio

---

## 2. STEP ENGINE (Execução)

Pipeline baseado em steps:

* BaseStep
* StepEngine
* Context compartilhado tipado (UploadPipelineContext)

Pipeline atual:

1. CreatePresignedUrlStep
2. UploadStep
3. MergeAttachmentsStep
4. AttachStep

---

## 3. RESILIÊNCIA (NÍVEL ENTERPRISE)

Implementamos:

### ✔ Retry por step

* max_retries configurável
* controlado por StepEngine

### ✔ Exponential Backoff

* delay dinâmico

### ✔ Jitter (estilo Netflix/AWS)

* evita retry storm

### ✔ Retry condition

* retry apenas para erros relevantes

---

## 4. CIRCUIT BREAKER

Implementação completa:

Estados:

* CLOSED
* OPEN
* HALF-OPEN

Comportamento:

* bloqueia chamadas após falhas consecutivas
* reabre após timeout
* isolado por tipo de step

---

## 5. CONFIGURAÇÃO DINÂMICA

Criamos:

* UploadConfig
* RetryConfig
* CircuitBreakerConfig

Permitindo:

api.files.upload(
request,
config=UploadConfig(...)
)

✔ comportamento configurável pelo usuário
✔ sem hardcode
✔ nível SDK profissional

---

## 6. PIPELINE ORIENTADO A CONTEXTO

UploadPipelineContext contém:

* request
* client
* card_service
* integration
* config
* upload_url / download_url
* files

✔ elimina uso de dict

✔ previsibilidade

# 🧠 ESTRATÉGIA DEFINITIVA PARA UPDATE DE CAMPOS

## ❗ PROBLEMA

A API do Pipefy é:

* inconsistente
* não padronizada
* muda comportamento por tipo de campo

---

## 🎯 SOLUÇÃO

> 🔥 DISPATCHER POR TIPO DE CAMPO

---

## 📦 DESIGN

field_type → handler → estratégia de update

Exemplo:

{
"text": UpdateTextFieldHandler,
"number": UpdateNumberFieldHandler,
"select": UpdateSelectFieldHandler,
"attachment": UpdateAttachmentFieldHandler,
}

---

## 🧠 RESPONSABILIDADES

### Dispatcher

* decide qual handler usar

### Handler

* conhece comportamento do campo
* aplica regra correta
* usa CardModel

---

## 🔥 PRINCÍPIOS

✔ NÃO usar mutation genérica
✔ NÃO confiar na documentação
✔ SEMPRE validar comportamento real
✔ SEMPRE usar models

---

## 🔁 PADRÃO DE EXECUÇÃO

1. Buscar CardModel
2. Identificar field_type
3. Selecionar handler
4. Handler calcula novo valor
5. Executa updateCardField

---

## 🚀 EVOLUÇÃO FUTURA (já prevista)

* fallback por tipo de campo
* comportamento específico por field_type
* possível reutilização de pipeline + rules
